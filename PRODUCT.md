# Product — guizang-social-card-skill

這份檔案講**為什麼**。事實在 `HANDOFF.md`。

---

## 1. 這個 skill 解決什麼問題

**真正的問題不是「畫圖」，是「把一篇文章變成一組能打的 Instagram 圖」。**

普通使用者的現實鏈路：

1. 寫完一篇文章（筆記 / 影評 / 攻略 / 週記 / 教學）。
2. 想把它發到 Instagram 或 Facebook，需要 5-9 張圖 + 1 張封面。
3. 沒有設計師，沒有時間在 Canva / Figma 裡逐張拉。
4. 用模板套，又被幾百個相似封面擠掉點選。
5. 讓 AI 生成，結果大部分 AI 出圖就是「標題放正中 + 漸層背景 + 一個 emoji」。

我們要在第 2-5 步之間插一個東西，讓產出不像 AI，又不需要使用者懂排版。

### 核心矛盾

- **品質**：要像 Guizang 這種磁帶、墨水、左對齊網格的雜誌感，不能像 PPT 模板。
- **效率**：要 30 分鐘內能拿到可釋出的圖，不能讓使用者在 prompt 上反覆打磨。
- **不假**：資料、報價、版本號不能編。文字壓圖不能壓到人臉上。

三者之中**品質是分水嶺**。Canva 模板已經解決了效率。AI 出圖已經解決了速度。但都沒解決「看一眼就知道是用心做的」這件事。

## 2. 產品定位（一句話）

> 把一篇文章交給 Claude，30 分鐘內拿到一組可以直接發的 Instagram 貼文圖或社群封面，**長得像一本被偏愛的雜誌**，而不是套模板。

這裡「被偏愛的雜誌」是一個具體的視覺錨點：

- 上海譯文出版社的封面
- The New Yorker 的內文版式
- Apartamento / Kinfolk 的留白節奏
- Massimo Vignelli / Helmut Schmid 的瑞士網格

不是「能看」或「好看」，是**能讓人停下手指**。

## 3. 目標使用者與典型場景

主要輪廓是 **「內容創作者 + 工程師 / 產品 / 設計師 / 自由工作者」**。共同點：自己有內容，沒有「團隊裡專門做封面的人」。

| 場景 | 使用者型別 | 輸入 | 期望輸出 |
| --- | --- | --- | --- |
| 週報 / 工具週記 | 工程師 / 產品 | 一段 Markdown + 數字（小時數 / 價格） | 5-7 張 Instagram 4:5，資料圖 + 總結 |
| 旅行 / outdoor | 攝影愛好者 / 旅人 | 一組照片 + 文字日誌 | 一組貼文 + 限時動態組合（4:5 貼文 + 9:16 限時動態，同一篇文章的兩個尺寸）+ 5-9 張 Instagram 組圖 |
| AI / 產品 release | 創業者 / DevRel | 一段更新說明 + 截圖 | 一組 Swiss 風的 release deck |
| 影視 / 遊戲筆記 | 影評 / 遊戲 podcast | 一段評論 + 主視覺 | 1 張大圖壓字封面 + 資料榜單 + 總結頁 |
| 美食 / 食譜 | 美食部落客 | 文字步驟 + 成品照 | 4:5 教學卡（使用者提供照片） |
| 健身 / 居家 / 穿搭精選 | 生活風格部落客 | 推薦清單 + 產品圖 | 推薦卡 + 單品圖 |

**反向輪廓**（這個 skill 不服務他們）：

- 想做 OOTD / 日常自拍：我們不生成人像照片。
- 想做菜品大片擺盤：我們不替代專業美食攝影。
- 想做「夢核 / Y2K / kawaii」：和我們的兩套視覺系統正面衝突，硬接出來一定難看。

這些在 `references/category-cookbook.md` 裡寫明瞭「不接」，是產品決策，不是技術缺口。

## 4. 使用者視角的核心思考

整個工作流是按「使用者腦子裡的步驟」組織的，不是按「程式碼模組」組織的。

| 使用者腦子裡 | Skill 對應步驟 |
| --- | --- |
| 「我有這篇東西，想發 Instagram」 | Step 1 Intake |
| 「我該出幾張？寫什麼標題？」 | Step 2 Extract Story + Step 4 Plan Pages |
| 「我要這種感覺的圖，不要那種」 | Step 3 Choose Style Mode |
| 「隨便先出一版我看看」 | Step 4.5 Seed + Step 5 Build |
| 「這張圖有問題，再改改」 | Step 5 重新渲染 + Step 7 Deliver |

每個步驟都有一個明確的「使用者在意什麼」：

- **Step 1 不要審訊式提問**。能從已有資訊推斷就推斷，只問真的會改變結果的問題（平臺、比例、必須出現的元素）。
- **Step 2 是產品差異**。一篇 1000 字的筆記，應該是 5 張圖還是 9 張？哪些段落該被丟進文案區不放進圖？大部分 AI 工具不做這一步——這是為什麼它們出的影象「被切碎的文字」。
- **Step 3 不讓使用者選「主題色」**。讓使用者選「感覺」：是雜誌感的人文敘事，還是瑞士的工程冷感。色板是結果，不是入口。
- **Step 4.5 是體力活的轉捩點**。v0.4 之前每次任務都要從零寫 HTML，能寫出來，但一致性差。種子模板把「雜誌的形」固化下來，讓 AI 把精力花在「內容如何變形」上。
- **Step 6 文字壓圖**是最常翻車的位置。v0.5 之前 AI 經常把標題壓在人臉上，看起來不假但很難看。`image-overlay.md` 強制三步走（讀圖 / 寫主體對映 / 選 object-position）後才把這塊兜住。

## 5. 關鍵設計決策（為什麼這樣做）

### 決策 1：拆兩套視覺系統，不做「一站式」

更通用 = 更平庸。我們決定**只做兩件事**：

- **Editorial Magazine × E-ink**：serif + 墨水 + 雜誌結構（ledger / marginalia / pull quote / photo well）
- **Swiss International**：Inter + 一種 accent + 工程結構（matrix / KPI tower / h-bar / 編號 statement）

不做「卡通 IP 風」、「少女風」、「國潮」、「Y2K」。一個 skill 同時維護 5 種風格的結果一定是 5 種都做不好。

**注意**（v0.10 起明確）：兩套系統是**視覺立場**，不是**內容型別門**。一篇 AI 筆記可以走 Editorial（如果你想要它讀起來慢、像專欄），一組旅行支出也可以走 Swiss（如果你想要它讀起來像資料包表）。我們曾在 v0.9 之前在檔案裡把「Editorial = 人文 / Swiss = 工程」寫得過死，結果出現「美食教學一定要塞進 Editorial 哪怕使用者想要 step 資料感」這種錯誤路由。v0.10 解綁：選風格按「你想要這篇內容讀起來什麼節奏」，不按「這篇是哪個品類」。代價是使用者多一道選擇題，收益是別再讓風格綁架內容。

代價：使用者如果想要的「夢核氛圍」在我們能力圈外，要在 Intake 時直接告知，不能硬接。這是產品邊界，不是 bug。

### 決策 2：種子模板 = 看不見的設計師

v0.4 之前，每次任務 AI 都從空檔案開始寫 CSS。結果是：

- 字型堆疊每次都重新寫，偶爾忘記加 fallback。
- 間距 token 每次都重新發明，導致兩張圖之間間距不一致。
- 卡片填充邏輯混亂，Swiss 的「四類卡片互斥」規則形同虛設。

種子模板把這些**藏在使用者看不見的地方**。使用者拿到的產出依然是「一份 HTML」，但 AI 不用再每次重新決定「是用 `--gap-7` 還是 24px」。這是為「品質上限」和「一致性」花的成本。

### 決策 3：components.md 的硬上限是「產品承諾」

`.h-xl` 在 Instagram 4:5 板上 96px、最多 2 行、單行最多 8 個漢字。這不是技術上限——CSS 當然允許寫 120px、3 行、12 個字。這是**產品承諾**：

> 我們承諾出的圖不會讓你尷尬。
> 如果你的標題撞破這個上限，AI 會主動建議你壓縮文案，而不是悄悄把圖拉變形。

很多 AI 工具的失敗模式是「使用者說什麼就硬塞什麼」。我們選擇反過來——**告知使用者文案太長，比交付一張醜圖更禮貌**。

### 決策 4：圖片來源「先取後說」

`SKILL.md` Step 6 的 web-sourced 部分寫得很明確：

> Policy: 先取，後告知，讓使用者決定。不要靠猜版權預先過濾。

理由：

1. 大多數 AI 工具的「版權安全」邏輯實際是「假裝版權安全」。在 Unsplash / Wallhaven 取圖本身不構成侵權，**是否在終稿署名是使用者的判斷**。
2. 使用者是終稿的所有人。把決策權交給他，附上 `SOURCES.md` 留痕，是更尊重也更準確的做法。
3. 等「完美的免版權相簿 API」會餓死。先解決「Instagram 今天能發」，慢慢解決「版權完美」。

### 決策 5：暗色 Editorial 只開一套——Midnight Ink

v0.12 之前 Editorial 只有 5 套淺色 palette。Codex 在測試黑神話主題時自造了 `wukong-night`，事後看是合理外推（遊戲 key art 強行套淺色 paper 會塌），但 skill 檔案當時還在寫「用 5 套之一不要改」，造成「規則太死 vs. 視覺合理」兩難。

v0.12 解法：**新增官方暗色 Midnight Ink，但只開一套**。

- 開一套的理由：遊戲 / 夜景 / 影調封面這類內容確實需要暗背景，硬塞淺色 palette 是品質塌方。
- 不開多套的理由：暗色 palette 的差異（金輝 / 冷藍 / 紫羅蘭 / 血紅）很容易滑向「遊戲宣傳海報」的廉價感。一套金輝（`#d4a04a`）作為預設 accent，既呼應黃金樹 / 神道 / 唐宋夜畫這類經典暗調，也守住「雜誌感不是遊戲感」的底線。
- 使用者想要其他暗色 accent → 優先建議改用 Swiss 暗變體（未來可考慮）或回退選最接近的淺色 palette 重新構圖。

這是和「決策 1：拆兩套視覺系統不做一站式」同一類思路——**少而準 > 多而平庸**。

### 決策 6：交付前不自動跑 validator，問使用者再決定

v0.6-v0.11 都把 validator（`validate-social-deck.mjs`）寫成「渲完必須自動跑」。v0.12 改成「渲完先把圖給使用者看，問一句要不要核查，使用者說要才跑」。

起因是真實工作流裡的兩件事：

1. **校驗耗時**。一份 5 張圖的 deck 跑一遍 R1-R6 + 重新渲染修補，常常 5-10 分鐘。多數情況使用者的眼睛在 30 秒內就能掃出「哪張需要改」。
2. **校驗的盲區**。Validator 測的是溢位 / 字級下限 / 行 cap，**測不出**「這張圖氣質塌了 / 這個 ledger 第三條沒說人話 / 這個 accent 在暗背景上太蠟黃」——這些是使用者最在意的、也是 AI 最容易翻車的位置。

所以讓使用者做第一道審查、自動校驗作為兜底的可選項，而不是反過來。代價是「完全自動化」的承諾降級——但這個 skill 的目標從來不是無人工，是**30 分鐘內出圖**。校驗時間從預設路徑裡摳出來，能直接讓平均交付時間從 8-12 分鐘壓到 3-5 分鐘。

### 決策 7：所有圖必須是靜態 PNG，沒有動效

社交平臺匯出後只顯示靜態圖。任何 motion / animation / WebGL 互動在最終產物裡**看不見**。所以：

- 不引入 Motion One / Framer Motion
- 不寫翻頁 / 鍵盤導覽 / ESC 索引（這些是 PPT skill 的需求）
- WebGL 背景只用於一次性渲染紋理，不真的「動起來」

這條決策讓 skill 比同等複雜度的「PPT 網頁」輕一個數量級。

## 6. 反模式（我們主動拒絕的事）

檔案裡散落著多處 anti-pattern。這裡集中歸納一下：

1. **不可讀的小字**：正文低於 26px on 1080×1350 → 拒絕。
2. **重字級 + 重字重的標題**：Swiss 80-120px 配 weight 700 → 拒絕。「越大越細」是 Swiss 的核心。
3. **任意裝飾 SVG**：blob、雨滴、kawaii 貼紙、隨手畫的圈 → 拒絕。
4. **巢狀卡片 / SaaS 拼盤**：預設版面不應該是「卡片裡再放卡片」→ 拒絕。
5. **裸跑圖片壓字**：照片沒有 quiet zone、標題壓過主體，或縮圖不可讀 → 拒絕；區域性 tint 只是兜底，不是預設遮罩。
6. **假資料 / 假版本號 / 假報價** → 拒絕。
7. **一圖複用到所有比例**：把 9:16 直接裁成 4:5 → 拒絕。兩個比例必須分別構圖。
8. **圖上加教學說明**：鍵盤快捷鍵、操作步驟壓在卡上 → 拒絕（教學內容寫在 caption / 文案區）。

每一條都對應了線上觀察到的「AI 出圖很糟」的具體形態。這些拒絕是 skill 的**護城河**。

## 7. 未來想做的事

按 ROI 排序。下面分兩層：**短期已規劃** 和 **中長期會做**。最末尾只有一條是真的不做。

### 短期已規劃（下兩輪內開工）

1. **Editorial 模板的冒煙測試**。v0.6 只在 Swiss 上跑了真實小例子，Editorial 模板尚未做同樣的硬化。預計會暴露 M14 縱向 Pipeline 5 步在 4:5 上的溢位，以及 M11 Marginalia Essay 的字級上限。下一輪優先做。

2. **校驗指令碼 `validate-social-deck.mjs`**。指令碼已完成，預設作為使用者要求 auto-check 時的兜底：
   - 檢查每張圖的 `scrollHeight` 是否超過 board 高度
   - 檢查正文字級是否低於平臺最小值
   - 檢查 `.poster.post` 上的 `.h-xl` 是否超過硬上限
   - 後續可補媒體左對齊 / browser-default margin 這類視覺漂移檢查
   把「渲染前看不出來 / 渲染後才發現溢位」這種坑用指令碼兜底，同時不阻塞使用者先看圖。

3. **一次出多平臺的「包」**。目前一篇文章如果同時要發 Instagram + Facebook，要開兩個任務資料夾各做一遍。可以做一個 `deck` 概念：一份 source content → 同時產出 **Instagram 5-9 張 4:5 組圖** + **貼文 + 限時動態組合（4:5 貼文 + 9:16 限時動態，同一封面的兩個尺寸）** → 一次渲染出全部 PNG。
   關鍵設計點：避免把 9:16 內容直接縮到 4:5（產品決策 7 已禁止）。需要在 plan 階段就為 9:16 限時動態寫直式分行的標題（上三分之一）、為 4:5 貼文寫一版短標題。

4. **4:5 貼文短標題生成器（貼文 + 限時動態組合的同源短標題）**。9:16 限時動態用「一週的程式碼 · 我和五個 AI 工具的同居生活」（直式分行放上三分之一），4:5 貼文用「AI 同居生活」。注意這是**同一篇文章的貼文 + 限時動態組合**，不是組圖——9:16 限時動態有垂直空間可分行放較長標題，4:5 貼文空間更緊湊所以必須壓短。目前兩個尺寸的標題靠 AI 自由發揮，可以加一個明確的「6-8 字短標題」約束在 Step 2 的 plan 階段。

5. **地圖元件**。旅行類內容經常需要「路線圖 + 關鍵節點」。真實攻略地圖預設用 Mapbox Static；沒有 token 時用 OSM 靜態圖磚；SVG schematic 只用於抽象關係圖。
   **執行路徑**：用 Mapbox Static Images API 或 OSM tile composite 出 1080×720 / 1080×1350 PNG，限制底圖為中性淺色 / 暗色以匹配 Editorial / Swiss，禁用花俏色板。節點用 Swiss 編號風（黑底白數字）或 Editorial 墨點風，避開預設藍針。GeoJSON 僅接 LineString + Point，複雜面／標註超出範圍。
   **路由**：先接入 `category-cookbook.md` 的「旅行」路由，作為 M02 / S08 的可選 inset；不強推。

6. **截圖美化資產（從 PPT skill 借現成的）**。Tutorial / News 類目最大的痛點是「截圖本身不好看」——背景太雜、有陰影、文字太多。可以加一組工具：自動加圓角、自動模糊背景、自動裁安全區。
   **執行路徑**：`~/.claude/skills/guizang-ppt-skill/` 已經有現成的截圖處理資產，直接 port 過來即可（與 v0.4 拷 Swiss class 的方式一致——只搬截圖美化相關的部分，不帶 PPT 翻頁 / 動效那一坨）。
   **範圍控制**：只搬「加圓角 / 加陰影 / 加 device frame / 裁安全區」四件套，不帶 PPT 那邊的多窗格拼貼；社交卡是單螢幕閱讀，截圖最多並排兩張。

### 中長期會做

7. **品類專屬配圖 prompt 庫**。`category-cookbook.md` 已經路由「什麼品類 → 用哪些 recipe」，下一步是「什麼品類的封面圖應該用什麼 prompt 去生成」。outdoor 需要「晨霧 / 冷調 / 35mm 底片感」，AI 產品需要「keyshot 渲染 / 極少元素 / 單光源」——這些 prompt 現在散在每個任務裡重寫。

8. **Instagram 1:1 方形貼文的種子**。目前兩份種子主打 4:5 + 9:16。1:1 是次要適配。如果 1:1 方形貼文格式起飛，需要一份獨立的 1:1-first 種子。

9. **AI 出圖後處理**。Gemini / Imagen 生成的圖經常有「AI 印記」（多手指、變形字、不自然光照）。可以加一步「AI 出圖 → 用 Read 工具讀 → 標註問題 → 給使用者決定要不要重出 / 區域性重繪」。
   **執行邏輯**（重點，因為分析圖片耗時）：
   - 預設**渲染完成後不自動分析**，避免每次任務多一次幾十秒的等待。
   - 渲染完成後在交付訊息裡主動問一句：「這頁圖裡 AI 生成了 `assets/hero-xxx.png`，要我幫你檢查一下有沒有 AI 印記嗎？需要幾十秒。」
   - 使用者說要 → 用 Read 工具讀圖、給出問題清單（多手指 / 文字變形 / 光照斷裂）、按問題嚴重度建議「接受 / 區域性重繪 / 整張重出」。
   - 使用者說不要 → 跳過，正常交付。
   也允許在 Intake 階段就指定「出圖後預設幫我檢查」，讓需要嚴格品管的使用者一次設定長期生效。

10. **影片 + 圖組合包**。讓 skill 出完 5 張圖之後，呼叫獨立的影片 skill（`seedance-prompt` / `video-wrapper`）順手出一個 15s 短影片，作為 Instagram 第一幀的「動態封面」。
    **執行邊界**：本 skill 不自己做影片生成，只做**排程** —— 把 plan 階段拿到的鉤子、調色、文案傳給影片 skill，避免重複一次內容理解。

11. **Web UI / 拖曳編輯**。Skill 的最終產物是 HTML+PNG，技術上可以做一個輕量的「出完圖後讓使用者在瀏覽器裡拖一下」的小編輯器（比如 React 包一層、改改文案／換圖／調間距、再觸發重新渲染）。
    **執行邊界**：不做「從零拼版」的 Canva 平替——那背離了「文字 → 設計」的核心鏈路。只做**渲染後微調**，作為最後一哩路。

### 真的不做

12. **風格自訂 / 使用者主題色**。誘惑很大，但一旦開放，品質會崩。目前 10 套 palette 是經過視覺驗證的上限。讓使用者傳一個 `#A3FF21` 進來，AI 配的衍生色大機率難看。除非有「色板生成器」先過一層，否則不開。
    這是一條**產品邊界**：寧可讓使用者從 10 套裡選一個最接近的，也不讓使用者精確指定。和「不接 OOTD / 夢核」是同一類決策。

## 8. 度量方向

目前沒有自動度量，全靠看產出。如果未來要量化，三個方向：

- **完成率**：一個真實小例子（5-9 張圖）的端到端用時。目標：30 分鐘以內（v0.6 的冒煙測試在 ~25 分鐘做完，含修兩次 overflow）。
- **首版可用率**：第一次渲染出來的圖，使用者不需要任何修改就能發的比例。目標：> 60%。
- **品類匹配命中率**：使用者說出 Instagram 品類後，路由到的 recipe 使用者接受的比例。目標：> 80%（不接受的需要進 `category-cookbook.md` 調整）。

這些都暫時是「心裡的目標」，沒有埋點。

## 9. 給接手者的兩句話

> 當你拿到一份內容，先問「這是不是我們能力圈內的品類」。
> 當你出完一張圖，先問「如果使用者截圖發給朋友，朋友會不會願意停下來看」。

前者管「不假」。後者管「能打」。兩件事都做到，這個 skill 就值得繼續存在。

## 10. 短影音（Reels／限時動態）復盤（2026-07-01）

短影音封面不是「把影片塞進卡片」。它是把原來靜態圖裡的一個 image well 換成 motion well，所以仍然要先遵守社交卡的版面、字型、配色、裁切和可見文案規則。

這兩天討論後，短影音封面的產品定位收斂為四件事：

1. **使用者素材優先**。正常使用者路徑是使用者上傳自己的影片、螢幕錄影、遊戲片段或生活素材。網頁找影片只用於我們自己做 demo / promo / 回歸測試，不應該被寫成使用者工作流。
2. **資訊量判斷優先於品類標籤**。品類庫不是固定模板表，而是判斷 `3s` / `5s` 內能展示多少資訊：一個動作、一個小過程、一個前後變化、三個並行結果，還是根本不適合做短影音封面。
3. **拼貼是正式能力**。當素材品質足夠好，可以少字甚至無字，把單影片、上下二格、上中下三格、四格作為 material-first 短影音封面輸出。單影片加字時使用 M16 / image-overlay 的圖片壓字規則；二格、三格、四格預設不加文字。
4. **三連短影音是高資訊密度玩法**。Instagram 流行的三列長圖式短影音，本質是把三個並行短片段放在一個卡片單元裡。它適合三種結果、三個視角、三步對比，不適合把一個長流程硬拆成必須順序閱讀的教學。

長影片處理不能承諾「精準自動找高光」。目前合理做法是低成本診斷：稀疏抽格 / contact sheet / 關鍵時間點預覽，用來發現候選段落，再讓使用者選擇修剪、倍速、拆成三連、或指定時間範圍。密集抽格能更準，但 token 和時間成本過高，不應該作為預設路徑。

釋出體驗只需要在交付時提醒清楚：

- Instagram 短影音封面預設按 `5s` 控制。
- Facebook 粉專貼文短影音封面按 `3s` 控制。
- 兩個平臺都應走手機端釋出路徑。iOS Live Photo（`.pvt`，iOS 專屬、非 IG 原生釋出格式）透過 AirDrop 發到 iPhone 後，從對應 App / 手機端編輯路徑釋出；電腦端 / 網頁端不能被當作可靠的短影音釋出入口。

這次討論串裡暴露的主要問題，不是渲染能力不足，而是執行者沒有把舊 Skill 當成硬約束：

- 把內部製作要求寫成觀眾可見文案，例如把「三連拼貼」、「5 秒看完」、「別硬塞」寫到卡面上。
- 在沒有模板依據時自造背景色、主題、橙色線條、刻度字、kicker、meta、hairline。
- 為了顯得「有排版」而新增額外元件，而不是用既有模板裡的字型、斷行、對齊、位置和留白完成排版。
- 把自動密度警告當成必須加裝飾的理由，忽略了 material-first 拼貼本來就可以少字或無字。

所以短影音封面的驗收標準要比「能動」更嚴格：第一幀必須像一張合格的靜態社交卡；運動只增強內容證據，不替代版面判斷；所有可見文字都必須像使用者真的會發給觀眾看的內容。
